How these numbers are calculated. Every Claude Code session writes a timestamped event log to disk. The build_dashboard.py script walks those logs event-by-event and measures the gap between each pair of consecutive events.

Claude runtime (22h 7m) is the sum of gaps that end in an assistant message or tool result — i.e. Claude itself was computing to produce that event. Capped at 15 minutes, since Claude rarely pauses longer mid-chain. This work ran in parallel with other projects.

Direction time (24h 6m) is the sum of gaps that end in a user prompt — reading output, reviewing diffs, and writing the next instruction. Capped at 30 minutes (gaps longer than that are treated as genuine breaks: lunch, meetings, next day).

Independently: Opus generated 3.8M output tokens at ~60 tokens/sec ≈ 17.8h of pure generation. Adding 5,124 tool calls (Bash, file I/O, subagents) lands in the 20-25h range — the log-derived 22h 7m matches, which is a good cross-check on the methodology.

What's working
Repeatable pipeline
Spec-driven section builds

The repo's scraped HTML + Figma references + per-component spec docs + three-round audit loop is a genuinely repeatable system, not ad-hoc prompting. Header, hero, footer, predictive search, collection pages, carousels — all shipped through the same workflow.

Documentation as handover
Docs ship alongside code

22 spec docs and handover markdown treat documentation as a first-class deliverable — decisions, deferred items, and client questions are captured in-repo. The engagement crossed 111 sessions without losing context, because the docs carried it.

Tight commit cadence
115 commits, mostly green

Small, frequent commits with the Shopify GitHub integration auto-syncing to staging meant changes could be reviewed in the browser within minutes of landing. 111 of 122 tracked outcomes fully or mostly achieved (~91%).

Strength areas
Multi-file refactors & debugging

Claude's top capabilities in this project: multi-file changes (52 sessions), debugging (25), explanations (23), and correct code edits (15) — matches a theme build where consistency across Liquid/CSS/JS is critical.

Where friction showed up
Design fidelity
Guessing visual details instead of verifying

The single biggest friction source: Claude fabricating arrow SVGs, progress bar colours, scrollbar styling, or border radii instead of checking the scraped HTML or Figma first. Required extra correction rounds on the featured collection swiper, filter sidebar, and collection pages.

Build pipeline
Wrong-file edits & rebuild wipeouts

Claude edited compiled theme.css instead of the Tailwind input.css in one session; a Tailwind rebuild during footer work wiped uncompiled CSS and needed an emergency hotfix. Both preventable with hooks or CLAUDE.md pinning.

Scope discipline
Jumping ahead of the spec

On a few sessions Claude started implementing before the spec doc was locked — leading to rework when the actual request was to update the doc first. The "scope-first" pattern works when stated explicitly at session start.

Root cause hunts
Investigating wrong hypotheses first

The Splide crash on product pages and the nested style-tags bug both saw Claude chase the wrong root cause before landing on the real fix. Faster resolution when asked upfront to enumerate 2-3 hypotheses before picking one.

Suggested next moves
Automation
Codify the section-build flow as a Skill

Every section followed the same shape: check scraped HTML → check Figma → write Liquid/Tailwind → validate → commit. Wrap that into a /section skill that asks for the Figma URL upfront, forces the verification step, and refuses to touch compiled theme.css.

Safety
Add PostToolUse hooks

A PostToolUse hook that (a) blocks writes to assets/theme.css, (b) runs shopify theme check after any Liquid edit, and (c) validates section JSON would have caught both the CSS wipeout and the stray-brace incidents before they reached main.

Stretch goal
Figma-to-Shopify pixel-match loop

Combine the Figma MCP with a Playwright MCP: Claude builds the section, screenshots the staging store, diffs against the Figma frame, and self-corrects up to N rounds. Turns the multi-round refinement cycles we saw in this engagement into hands-off builds.

Governance
Extend the spec library across clients

The 22-doc library here (build-playbook, foundation, header, footer, product-page, etc.) is a template other eComplete theme builds could inherit. Publishing it as an internal starter repo would shortcut the Phase 0 scope work for the next engagement.

Skills Used

/audit
Triple-round reference audit
Runs three parallel agents against scraped HTML, repo source, and the spec doc. Flags markup, style, or behaviour inconsistencies before a single line of theme code is written.
🏗️
/build
Spec-driven component build
Reads the component spec, scraped HTML, and reference React/TSX. Writes Liquid, Tailwind classes, and vanilla JS in the exact style of the repo — no framework drift, no invented settings.
✅
/validate
Theme + browser validation
Runs Shopify's theme validator via MCP, checks schema correctness, then drives the staging storefront to verify the feature renders and behaves against the scraped baseline.
📦
/handover
Session handover generator
Scans the session log and diff, emits a handover markdown: files touched, decisions made, deferred items, and links into the spec docs for continuity across builds.
